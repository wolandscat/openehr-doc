= Generated UML and Specifications

== Website Contents

This website contains an automated rendering of the openEHR models, from source files expressed in BMML (BMM Language), a more advanced form of the BMM language in use in the openEHR specifications.

Currently, the main openEHR RM, BASE and LANG models are represented.

== Source Files

The BMML source files may currently be found in https://github.com/wolandscat/openehr-bmm[this Github repo^]. The form of BMM Language used in these files is reminiscent of a programming language, and includes a full representation of functions, invariants, pre- and post-conditions, along with build in void-safety and other features.

The source files can be edited within IntelliJ with a plugin that performs the usual functions of refactoring, inheritance visualisation and so on.

=== Specification / Package Structure

The top level packages are informally treated as 'specifications' on the generated site. In most cases, these are 1:1 with existing openEHR specifications. However, the package structure follows the new SPLASH design, not the openEHR component or model structures. In this approach, the following top-level components are define.

Model components:

**representation**:: all fine-grained representational elements, e.e. data types, structures, etc. Roughly equivalent to openEHR::BASE plus lower level parts of openEHR::RM.
**foundation**:: models of real world things, such as demographic entities, physical entities (inventory), investigations, etc. Reusable across many domains. Roughly equivalent to higher levels of openEHR::RM.
**domain-xxx**:: top-level models of particular domains, that include domain-specific items (e.g. 'EHR' in healthcare is a particular kind of 'record' defined in `foundation`), as well as specific inclusions from the `foundation` level.

Support components:

**languages**:: formal language meta-models. Currently for openEHR this includes BMM, AOM, and BEL. AQL will be added at some point. Equivalent to openEHR::LANG.
**infostructure**:: generic technical elements such as versioning.

=== Model Maintenance

Editing is best performed in an IDE, currently IntelliJ, for which there is a BMM plugin supporting parser-based syntax highlighting, navigation (ctrl+B, ctrl+N), refactoring, inheritance visualisation (ctrl+H), and intelligent commenting. The plugin is under development and will be released soon.

== Toolchain

The site is built using http://https://antora.org/[Antora^], from `.adoc` (Asciidoc) source files, generated by a BMML processing tool (soon to be openly released). The publishing part of the tool synthesises UML diagram as PlantUML source. The stages of processing are as follows:

```
openEHR-bmm ----> (bmm_tools) ----> openEHR-doc -----> ( Antora ) ----> openEHR-doc-site
                  (publisher)                          (PlantUML)
BMM source                        Asciidoc source      ( kroki )           Website
                                 including PlantUML
                                       source
```

== About the UML Diagrams

The PlantUML source of the diagrams is fully machine-generated, so are formally correct (barring any bugs in the BMML UML publisher).

However, the layout is unavoidably somewhat uncontrollable, since it relies on GraphViz algorithms to try to optimise placement of the elements. There are many tricks for hinting to PlantUML how to do a better job, but these are challenging to apply in an automated generator. We are continually working on this aspect of UML generation.

== The BMML Formalism

An advanced form of the {openehr_bmm}[BMM specification^] is used to represent the model sources and to process them. It implements:

* functions (xref:infostructure:versioning-gen.adoc#_class_versioned_object[example,window=_blank]);
* lambdas, i.e. objects representing function calls, with function-currying (xref:languages:aom2-gen.adoc#_class_c_object[example,window=_blank] - see c_conforms_to());
* full generic typing;
* all expressions are fully computable;
* pre- and post-conditions;
* class invariants (xref:representation:data_marking-gen.adoc#_class_locatable[example,window=_blank]);
* literals of primitive types and of typical containers, i.e. `List<>`, `Map<>`, which may be nested, and easily map to JSON (xref:languages:aom2-gen.adoc#_class_c_temporal_definitions[example,window=_blank]);
* full code, which can be used to provide necessary logic, as used in the openEHR AOM spec (https://github.com/wolandscat/openehr-bmm/blob/main/bmm/languages/org.openehr/aom2/constraint_model/complex/C_ATTRIBUTE.bmml#L94[example^] - not yet rendered);
* meta-model extensions.

One effect of these features is that all of the semantics of a specification can now be computably represented, rather than just being documentation, for example complex constants and algorithms. This means that all specification semantics are reprocessable into other languages, representations and so on, as well as being directly executable, given a BMM virtual machine.
